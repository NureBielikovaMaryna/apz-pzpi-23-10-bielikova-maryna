Тема роботи: Програмна система для моніторингу громадського дорожнього руху 


Мета роботи:
В цій лабораторній роботі необхідно показати як можна масштабувати бекенд системи для роботи із великим навантаженням. 


1. Стратегія масштабування
Для даної роботи було обрано вертикальне масштабування шляхом декомпозиції монолітної системи на мікросервіси. Система була розділена на два незалежні функціональні вузли:
1. Auth Service (Порт 5152): Відповідає за аутентифікацію (Login, Register), профілі користувачів та збереження історії переглядів.
2. Trains Service (Порт 5153): Відповідає за розклад поїздів, керування станціями, маршрутами та оновлення затримок (Sensor Data).
Такий підхід дозволяє масштабувати кожен сервіс окремо. Наприклад, якщо кількість запитів на перегляд розкладу зростає, ми можемо виділити більше ресурсів тільки для Trains Service, не навантажуючи модуль реєстрації користувачів.


2. Технічні рішення
Для реалізації масштабування було впроваджено такі рішення:
1. API Gateway (Client-side): Фронтенд (React) виступає у ролі інтегратору, звертаючись до відповідних сервісів за різними адресами (AUTH_API та TRAINS_API).
2. Shared Database (Спільна БД): Сервіс використовує спільний файл бази даних SQLite, розташований за абсолютним шляхом на диску D:. Це забезпечує цілісність даних при їх зміні адмін-панеллю в одному сервісі та читанні клієнтом в іншому.
3. CORS Policy: Налаштовано політики доступу для обох серверів, що дозволяє крос-доменні запити з різних клієнтських портів.
4. Dev Tunnels: Для тестування розподіленої системи в мережі використано два незалежні тунелі, що дозволило перевірити роботу системи на поза локальною мережею.


3. Аналіз «вузьких місць» (Bottlenecks)
При навантаженні системи першими вичерпуються такі ресурси:
1. Database Locking (SQLite): Оскільки SQLite використовує блокування всього файлу при записі, одночасні запити від Auth Service (запис історії) та Trains Service (оновлення затримок) можуть створювати черги та затримки (Latency).
2. Network Bandwidth (Dev Tunnels): Пропускна здатність безкоштовних тунелів обмежена. При великій кількості запитів ($RPS > 100$) швидкість відповіді сервера суттєво падає.
3. Connection Pool: При великій кількості одночасних підключень до бази даних на одному диску може виникнути затримка дискового вводу-виводу ($I/O$).


4. Навантажувальні тести та результати
Я зосередилася на забезпеченні Consistency (цілісності) даних між мікросервісами через спільну БД та перевірці відмовостійкості. Для моєї системи критичним було показати, що при падінні сервісу розкладу, авторизація продовжує працювати, що і є головною перевагою масштабування
Замість автоматизованих інструментів було проведено тестування на відмовостійкість та ізоляцію сервісів:
Сценарій 1: Сервіс активен. Система витримує паралельні запити з веб-сайту.
Сценарій 2: Сервіс Trains Service (5153) було примусово зупинено.
Результат: Користувач все ще може увійти в систему, зареєструватися та переглянути свою історію через Auth Service (5152).
Висновок: Система масштабована вертикально — падіння одного модуля не призводить до повної зупинки бізнес-процесів (аутентифікація залишається доступною).


Висновок
У ході роботи було реалізовано перехід від монолітної архітектури до мікросервісної. Це дозволило ізолювати критичні вузли системи (авторизацію) від динамічних вузлів (розкладу), забезпечивши вищу стабільність та можливість незалежного масштабування компонентів.
